The Universe: 给开发者的 全新应用世界

这里是通向新Web世界的入口...

   

       标准的Web页面，已经缔造了一个丰富多彩的互联网世界，其丰富的表现力使得桌面应用显得相形见绌， 其灵活、易于制作等诸多优势也充分的展现了互联网的威力。不容否认，互联网浏览器是形成这一切的推动者；如何在桌面应用系统之上复制这个模式？ 这个考虑正是我们的切入点。将互联网浏览器编译成桌面应用的可加载模块，这样就可以延续Web的表现力，为此我们需要一种新的兼容于标准页面的WebPage……

       如果我们需要一种新的页面，那么如何体现这个“新”？让我们看看数学的发展是怎样的一种模式。我们知道，最初的人类， 应该只有自然数的概念，随着社会的发展，人们需要物质交换，进而产生了“分数”、“负数”等基本元素，这样数的概念被扩展了，现代代数理论基于此形成了“代数数论” 这个优雅的理论，其基本要点是：为了使得某一类代数方程“可解”，人们必须遵循一种“代数扩张”的原则，使得对应的方程的解可以存在，从有理数到无理数、由无理数 到复数，都可以按照这类规范逐步形成。我们知道，一系列经典的桌面客户端开发技术，已经形成了大量的组件，例如：ActiveX控件、.NET控件、C++View等等， 能否将其逐一加入到Web DOM，使得页面更加“应用化”？

       数学发展的历史，其实就是一种“扩张”史，为了满足某种需要，人们不断的增加新的元素，“数”的发展史是这样，微分方程也是这样， 为了使得某类微分方程是可解的，人们发明了“广义”函数；纵观数学的发展，可以说没有“扩张”的概念，就不会有今日的数学。回到互联网大行其道的今天，桌面应用与Web应用 之间明显的存在一个“鸿沟”，由于这个鸿沟的存在，我们不得不面对“桌面”与“Web”两个世界，从桌面应用的角度看桌面应用缺失很多东西，显得非常的“尴尬”……

       Web页面的“扩张”，相当于代数数论中的代数域扩张，我们需要一种新的页面技术，使得那些可编程组件可以“友好”的成为DOM元素， 同时不破坏Web技术赖以支撑的“沙箱”机制，这一点是The Universe的核心问题，随着可编程组件不断的融入DOM，应用页面的表现力就会变得越发生动，由于兼顾了“Web”与“桌面” 的双重优势，因此一个全新的Web帷幕逐步拉开了……

       The Universe本质上是一种“Web扩张”技术，其基本要点是将一切都看作“DOM”，或者“DOM元素形成的页面”，针对不同的场景， 人们需要不同的页面，以此来确保应用系统的互联网弹性。为此，我们需要改良标准Web页面的作用域，我们需要一种广义的“浏览器”，可以说WinForm窗体、MFC、Eclipse View等等 都可能是WebPage的加载器，都是某种意义上的浏览器，为此我们需要借鉴数学上的某种抽象能力，需要将传统意义上根本不是页面的对象（例如WinForm）看作为一种页面， 颇有“波粒二象性”的意味，即：任何一种可编程组件本身就具备“Web”与“桌面”双重特征，只不过目前我们仅仅挖掘了其中的一个侧面，德布罗意的“物质波”是打破观念的一次巨大推动， 现在需要我们用类似的观念来统一“桌面”与“Web”，只有这样，我们才可能让经典得以延续……

       基于互联网的应用生态概念就是建立在这样的背景之下：我们首先看到了Web页面的表现力、灵活性、易于发布等诸多优势；我们也 注意到了桌面软件庞大的技术积累，这两个方面恰好是“本地”与“网络”两个侧面，如何将二者的优势恰到好处的结合，我们认为这个结合点是形成互联网应用生态的关键。我们的具体 思路是，将标准的DOM元素、COM组件、.NET组件、C++组件甚至考虑到Java组件合并为一个巨大的“组件空间”，看看能否有一个更好的页面合成规范，这一考虑方式很接近于近代数学的 基本方式，在数学里，人们习惯于将所有的多项式（基于某个域的多项式）放在一起，由此形成的数学理论是极具典型的理论，例如，代数几何就是这样的出发点发展起来的；类似的思维 在数学理论里面是司空见惯的。那么，如果我们考虑一个小一点的模型，例如，将Web DOM与所有的.NET控件放在一起会怎么样？以此为“基”，会形成什么样的结构？虽然这样的考虑 作用域相对小，但也会促进一种页面的形成，The Universe就是类似的这样的出发点，我们充分的评估了Windows平台上的几大类典型的组件，以此形成了一类应用的公共基础……

应用页面概念的形成，或许会彻底改变人们组织桌面应用的方式...
       从组织方式上看，应用事实上就是一组页面，其第一个页面或许就是应用系统形态的具体体现，不同于传统的桌面系统，大多数情况下，传统桌面系统的第一个窗口差不多决定了应用类型， 例如，如果你的第一个窗体是WinForm，那么我们基本可以断定这个系统是.NET应用，现在的情况是，你的第一个页面是由页面描述决定的，这一点很像线性微分方程或者线性代数的线性方程组， 通常情况下这类方程都有无穷多解，有通解、特解之说，例如，偏微分方程，我们通常需要边值条件、初值条件才能够确定有意义的解，类似的，一个特定的应用系统，一般说来需要一个或者几个初始化页面，这意味着，一个程序的启动窗口，会随着初始化页面的不同而不同……        我们知道，对.NET而言，通常一个dll库，可以包含任意数量的控件或者Form，如果这些元素都可以做为Web DOM元素，那么每一个Form或者控件会对应无穷多个具体的运行太， 以下示范给出了一个单一的MDI窗体，在Web页面描述之后，对应的四种运行时状态：

       一旦各种窗体、UI可以用DOM描述，那么“DOM Define Software”的基本设想就会逐步成为现实，我们将这个过程称之为“应用页面化”。 由于页面可以在应用系统运营的过程中动态在家啊，以此，页面化赋予应用系统一个天然的“应用生态”，而且这个生态依赖的元素要远远多于标准的DOM，应用系统的弹性也就自然而然的形成了……


Dock: The attribute of being familiar and unfamiliar

Every .NET UI (WinForm, Usercontrol, WPF) object has a Dock property, which has always existed since the birth of the .NET Framework. Over the years, we have become accustomed to the mature use of the Dock attribute, and our thinking has fallen into inertia, forming a blind spot. Generally speaking, the Dock property divides UI objects into two categories, as shown in the table on the right.

About Dock Attribute Controls are classified by Dock Attribute

Because the Dock attribute is the second type of control, it must depend on a specific control. This dependency determines that this type of control generally cannot be the main bearing control of a certain type of content.

The first type of control includes the following two situations:

  • The control itself is a child object of another .NET UI control
  • Is Another Child Window of a non-.NET UI object (DockStyle.None)
